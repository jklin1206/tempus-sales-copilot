Scope & assumptions 02 / 08
What I built, and what I did not

The honest version of the scope.

Assumptions
  • 01"Provider" means the oncologistThe case study's word, and standard healthcare usage. The meeting is with a doctor or a CMO.
  • 02Population is a directional proxy"Ordered by impact" is asked for, but impact is not in the data. I sort by the one real signal and label it, rather than invent a composite score that looks clever and means nothing.
  • 03The prototype begins after ingestionCommitted snapshots. No Salesforce sync, no live scraping, no database.
  • 04Product facts are public‑page onlySo no pricing or coverage fact exists, and a cost question routes to a human. An eval check asserts that gap stays a gap.
Deliberately not built
  • ✗A chat interfaceThe rep has fifteen minutes and already knows what they need. Making them prompt their way to it hands the work back to them and returns something different every time. One click, one brief, read in thirty seconds.
  • ✗"Why now"The three supplied sources are timeless. A defensible answer needs a dated catalyst feed. I built one and cut it rather than manufacture urgency.

The tradeoff: no RAG, no agent loop
34 facts fit in one request. At this size a retriever can only lose, by dropping the one fact that mattered. No embeddings, no vector store, and no retrieval failure mode to debug.
The model never picks a tool or decides to retry. Code does. Every run is reproducible, and every failure is a test rather than a mystery.
The seam is already drawn. Retrieval drops in behind the knowledge interface the day the corpus earns it. Nothing above it changes.
Trust 06 / 08
Grounding

Every claim is checked before the rep sees it.

  • ✓The CRM quote is verbatim
  • ✓Every number traces to a cited fact
  • ✓Qualifiers survive"Typically within 7 days" never becomes "within 7 days."
  • ✓No guarantee, competitor, or pricing claim

If a check fails, the brief is blocked. The rep is never handed a claim we cannot back.

Process & tools 07 / 08
How I arrived at it

Spec first. Deterministic layers. Then the model.

  • 01Wrote the brief contract and the validation rules before any code
  • 02Built everything that needs no model firstLoaders, the ranking, the validator. All testable without a single API call.
  • 03Added the model last, constrained by the validator it would be graded on
  • 04Iterated the prompt against an eval, not against tasteFour versions, each driven by a measured failure.

Validation 08 / 08
How I refined it

The eval told me what to fix.

  • 01Declare the right answer firstFor each of the eight doctors I wrote down what the copilot should do, and which exact fact it must cite, before generating anything.
  • 02Generate, and let the validator rejectAnything that misses the declared answer fails. No human judgement in the loop.
  • 03Read the failure, not the scoreNot "bad output." It says: you dropped "typically" from xf‑turnaround‑01.
  • 04Fix the prompt. Re‑run. Measure.Four versions, each driven by a named failure.

Four times, the harness was wrong
The model wrote

"This is a typical expectation, not a guaranteed turnaround."

The validator blocked it

for using the word "guaranteed."

The model was doing exactly what the rule existed to enforce, and got punished for saying the safe thing out loud. So I fixed the check, not the model. Negated phrases are now stripped before the prohibition runs. "We guarantee 7 days" is still blocked.

A validator you never correct is a validator you do not trust.
